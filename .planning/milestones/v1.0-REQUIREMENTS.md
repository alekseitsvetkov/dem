# Requirements: HLTV CLI v1.0

**Defined:** 2026-05-02
**Archived:** 2026-05-03
**Milestone:** v1.0 MVP
**Status:** ✅ SHIPPED

## Shipped Requirements

All v1 requirements completed and validated.

### CLI Foundation

- [x] **CLI-01**: User can run the compiled Go CLI and see JSON output for successful commands. — Phase 1-2
- [x] **CLI-02**: User can request command help without triggering HLTV network calls. — Phase 1
- [x] **CLI-03**: User receives structured JSON-compatible error details on stderr and a non-zero exit code when a command fails. — Phase 1-4
- [x] **CLI-04**: Future commands can be added without changing existing command handlers or JSON response types. — Phase 1-4

### HLTV Access

- [x] **HLTV-01**: CLI fetches public HLTV pages with timeouts and a configured user-agent. — Phase 2
- [x] **HLTV-02**: HLTV provider logic is isolated behind interfaces so tests can use fixture HTML or mock transports. — Phase 2-4
- [x] **HLTV-03**: Parser tests cover expected HLTV page structures for events, results, and match demo links. — Phase 2-4

### Events

- [x] **EVNT-01**: User can list Tier 1 HLTV events as JSON. — Phase 3
- [x] **EVNT-02**: Each event includes stable available fields such as event ID, name, date range, location, and source URL. — Phase 3
- [x] **EVNT-03**: User can limit the number of events returned. — Phase 3

### Results

- [x] **RSLT-01**: User can list completed HLTV match results as JSON. — Phase 3
- [x] **RSLT-02**: Each result includes stable available fields such as match ID, teams, score, event, date, format, and source URL. — Phase 3
- [x] **RSLT-03**: User can limit the number of results returned. — Phase 3

### Demo Lookup

- [x] **DEMO-01**: User can pass an HLTV match ID and receive a JSON response containing the match source URL. — Phase 4
- [x] **DEMO-02**: User receives a demo download URL when HLTV exposes one for the match. — Phase 4
- [x] **DEMO-03**: User receives a distinct outcome when the match exists but no demo link is available (DemoURL field omitted, exit code 0). — Phase 4

## Deferred to v2

| Requirement | Rationale |
|-------------|-----------|
| DISC-01: Match search by teams/date | Fuzzy search deferred for reliability |
| DISC-02: Filter results by team/event/date | Filtering beyond v1 scope |
| DISC-03: Filter events by year/status/location | Filtering beyond v1 scope |
| DOWN-01: Download demo files directly | User asked for link, not file transfer |
| DOWN-02: Choose output directory for downloads | File download deferred |
| PERF-01: Local caching for repeated requests | Performance optimization deferred |
